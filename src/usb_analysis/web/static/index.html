1. Rychlý start

              
    
                
  1. Nahraj PCAP. Použij Upload v sidebaru vlevo nebo přetáhni soubory přímo na nahrávací pole. Aplikace přijímá .pcap, .pcap00 až .pcapNN ve formátu LINKTYPE_USB_LINUX_MMAPPED (záznam z linuxového usbmon).
    2. 
                
  2. Po nahrání se rozsvítí zelená tečka v hlavičce a zobrazí se záložka Přehled se souhrnem (počet paketů, doba trvání, sessions, runy, kritické chyby).
    3. 
                
  3. Pro detailní pohled přepni na záložku Flow analyzer — to je hlavní zobrazení, kde uvidíš chronologii příkazů a odpovědí.
    4. 
                
  4. Pokud něco selhalo, otevři záložku Errors — vidíš seznam detekovaných chyb. Klik na řádek tě skočne přímo do Flow na příslušné místo.
    5. 
                
  5. Pro CI integraci stáhni JUnit XML ze záložky Export.
    6. 
                
  6. Verze aplikace + limity: klikni na USB Analysis v levém horním rohu hlavičky — otevře se modální okno s aktuální verzí, limity uploadu a stavem serveru.
    7. 
              

            


            
            

              
2. Co která záložka dělá

              

                
ZáložkaK čemu

                
                  
PřehledSouhrn capture: počet paketů, doba, runy, sessions, kritické/varovné counts. První zastávka po nahrání.

                  
PaketySurová tabulka USB paketů (technický pohled). Každý řádek = jeden URB Submit/Complete s metadaty (bus, device, endpoint, payload hex).

                  
Stream„Follow Stream" jako ve Wiresharku — ASCII překlad bulk komunikace. Vidíš text který tester poslal a co zařízení odpovědělo.

                  
Flow analyzerHlavní view: chronologie command ↔ response s detekcí timeoutů, CRC chyb, reconnectů, kauzálních souvislostí. Klik na řádek ukáže detail vpravo.

                  
ErrorsFiltrovaný seznam jen problémů (timeouts, CRC mismatch, EPROTO atd.). Klik = skok do Flow.

                  
SessionsPer-zařízení (bus/dev/serial) a per-run statistiky. Užitečné když capture obsahuje výměnu zařízení nebo víc runů testu.

                  
Deep analysisStrojové učení: porovnání proti naučenému baseline, mining pravidel napříč soubory. Dlouhá operace — spouští se ručně tlačítkem, ne automaticky při přepnutí na záložku.

                  
ExportStažení flow analýzy v JSON / CSV / HTML / JUnit XML.

                  
NápovědaToto okno — manuál, slovník event tříd a klávesové zkratky. Dostupné i bez nahraného capture.

                
              

            


            
            

              
3. Jak číst Flow analyzer

              
Každý řádek je jedna událost. Sloupce zleva:

              

                
SloupecVýznam

                
                  
seqPořadové číslo události v capture. Klíč pro odkazy a párování.

                  
delta_msDoba od předchozí události v milisekundách. Velký skok = potenciální zaseknutí.

                  
directionHOST→DEV tester poslal data; DEV→HOST zařízení odpovědělo; INTERNAL meta událost (timeout, reconnect…).

                  
dev-tagŠedý badge b1/d4|D736D92D = bus/device/serial. Pomáhá rozlišit, který USB zařízení událost vyprodukovalo.

                  
[event_class]Typ události — viz glosář níže.

                  
contentLidsky čitelný popis: pro command obsahuje text příkazu, pro chyby vysvětlení důvodu.

                
              

              

                Indikátory na řádcích:
                žlutý okraj = warning,
                červený okraj = critical,
                ⛓ = chunkovaný příkaz (rozdělený na víc paketů),
                žlutý levý proužek = causal suspect (tato událost je v kauzálním okně jiné chyby).
              


              
Toolbar nad seznamem

              

                
Ovládací prvekK čemu

                
                  
SeverityALL = vše včetně ok, info+, warning+ (výchozí), critical. Pozn.: normální command a OK response mají severity ok, takže ve výchozím filtru warning+ nejsou vidět — pokud chceš vidět běžnou komunikaci, přepni na ALL.

                  
DirectionFiltruje HOST→DEV, DEV→HOST, nebo jen INTERNAL meta události (timeout, reconnect…).

                  
RunČíslo runu — zobrazí jen události z konkrétního runu. Prázdné = vše.

                  
SearchFulltext nad content všech eventů. Hledá i v rozšířeném textu odpovědí.

                  
Skoč na #Zadej seq číslo + Enter (nebo klávesa G) — okamžitě skočí na řádek a vybere ho.

                  
← / →Historie navigace ve flow (skoky přes search, paired-seq, sessions…). Alt+šipky stejně.

                
              


              
Spodní timeline graf

              
Pod tabulkou je hustotní graf (bucket-y po čase): zelené sloupce = ok, žluté = warning, červené = critical, fialové = reconnect. Klikni do grafu pro skok na první událost v daném časovém úseku.


              
Detail vpravo + kauzální kontext

              
Klik na řádek zobrazí JSON detail vybraného eventu vpravo. Pod ním je Kauzální kontext — události, které dle analyzéru mohou souviset (jiné warningy/criticals v okně 10 eventů zpět). Klik na kandidáta tě tam přesune.

            


            
            

              
4. Slovník event tříd

              
Každý řádek ve Flow má tag jako [command], [timeout], [incomplete_segment]. Tady je co znamenají:


              
Tester SN vs. DUT SN — kdo je kdo

              

                
PojemVýznam

                
                  
Tester SNSerial programovací stanice (testeru). Pochází z odpovědí OK XXXXXXXX (8 hex znaků). Je stejný pro všechny runy v jednom capture — jeden tester programuje mnoho DUTů.

                  
DUT SNSerial konkrétního kusu hardware (Device-Under-Test). Pochází z příkazu checked-otp-device-sn-write <SN> — první argument za názvem příkazu. Mění se per run — každý run programuje jeden nový DUT.

                  
device_serial v event JSONAktuální identita: DUT SN má přednost (jakmile je v runu zapsán), jinak tester SN. Toto je „kdo právě teď komunikuje" z pohledu uživatele.

                
              


              
Komunikační události

              

                
TřídaCo znamená

                
                  
commandTester poslal příkaz na zařízení (např. ping, checked-optiga-init). Obsah řádku je celý odeslaný text včetně CRC suffixu.

                  
command_chunkPokračovací paket dlouhého příkazu (např. checked-secrets-certdev-write s velkým certifikátem se nevejde do jednoho USB paketu, posílá se po kouscích). Není to chyba — jen vizualizuje fyzické rozdělení.

                  
crc_probeTester zkouší jestli má zařízení zapnutý CRC režim. Záměrně pošle příkaz bez CRC. Pokud zařízení odpoví ERROR invalid-crc, znamená to že CRC je aktivní (= probe byl úspěšný).

                  
response_okZařízení odpovědělo OK nebo OK NO. OK NO = operace proběhla, ale výsledek je negativní (např. lock-check říká „nezamčeno").

                  
response_errorZařízení odpovědělo ERROR. Vždy critical (kromě crc_probe odpovědí, které jsou očekávané).

                  
response_progressPrůběžný status řádek (začíná #) — informativní, ne výsledek operace.

                  
response_dataOstatní data od zařízení, která nezačínají OK ani ERROR ani #.

                
              


              
Detekované problémy

              

                
TřídaCo znamená

                
                  
timeoutTester poslal příkaz a do 30 s nedostal odpověď. Zařízení se buď zaseklo, nebo odpověď nedorazila po sběrnici. Obsah řádku ukazuje skutečnou prodlevu (např. Timeout 35000.0ms na checked-optiga-init).

                  
incomplete_segmentPředchozí příkaz nebyl uzavřený a přišel nový. Příklad: tester poslal příkaz A, ale ještě než A dostal odpověď, poslal příkaz B, nastal reconnect, nebo se změnilo zařízení (device_change). Stav komunikace je nekonzistentní — zařízení může nebo nemusí příkaz A zpracovat. Pozn.: continuation paket chunkovaného příkazu (bez CR/LF terminátoru u parenta) je rozpoznán jako command_chunk, ne incomplete_segment — chunkování není chyba. Ale pokud chunkovaný příkaz přeruší device_change nebo timeout, incomplete_segment se emituje, protože odpověď už nikdy nedojde.

                  
reconnectMezi dvěma bulk přenosy byla mezera >5 s — pravděpodobně se zařízení odpojilo a znovu připojilo, nebo se restartovalo. Často následuje sekvence control transfers (re-enumerace).

                  
usb_errorUSB chyba na fyzické vrstvě (např. status -71 = EPROTO = chyba protokolu, signal integrity, DN/DP). Pokud následuje reconnect, je to typicky reset zařízení (info severity); izolovaná EPROTO bez reconnect = critical.

                  
lost_urbUSB Request Block byl odeslán (Submit), ale ani po 60 s nedorazil Complete. Zpráva se „ztratila" v ovladači nebo zařízení.

                  
device_changeV capture se objevil paket s jinou kombinací (bus, device_address) — typicky výměna zařízení nebo reassignment po reconnect. Zde končí jedna device session a začíná další.

                
              


              
Tagy v Errors tabulce (po detektoru)

              

                
event_typeCo znamená

                
                  
missing_crcPříkaz checked-* byl poslán bez CRC suffixu (8 hex znaků na konci). Zařízení v CRC režimu ho odmítne.

                  
crc_mismatchCRC suffix nesouhlasí s vypočítaným CRC32 obsahu příkazu. Buď chyba testeru, nebo poškození na sběrnici.

                  
app_errorAplikační vrstva: zařízení vrátilo ERROR ....

                  
lock_regressionPo úspěšném checked-*-lock následný lock-check vrátil OK NO (= „nezamčeno"). Něco se odemklo, což být nemělo.

                  
timing_high / timing_criticalLatence příkazu >5× / >15× vyšší než medián jeho ostatních volání. Zařízení něco zpomaluje.

                  
timing_suspiciously_lowLatence <50% nejmenšího pozorovaného minima. Možná zkratka v logice nebo cached odpověď.

                  
usb_eproto_isolatedEPROTO chyba bez následujícího reconnect — pravděpodobně permanentní hardware problém.

                  
usb_eproto_reconnectEPROTO chyba následovaná reconnect — typicky reset zařízení po watchdog timeout.

                  
usb_lost_urbViz výše — Submit bez Complete v okně 60 s.

                  
device_reconnectReconnect zařízení (může být očekávaný nebo neočekávaný v závislosti na kontextu testu).

                
              

            


            
            

              
5. Severity (závažnost)

              

                
ÚroveňCo to znamená

                
                  
okNormální stav (úspěšná odpověď, validní příkaz). Většinou skryté ve výchozím filtru warning+.

                  
infoInformativní (probe, progress, očekávaný reconnect). Není problém, ale stojí za zmínku.

                  
warningNěco není ideální (OK NO, vyšší latence, lost URB). Komunikace funguje, ale měl by ses na to podívat.

                  
criticalDefinitivní problém (timeout, ERROR, missing CRC, isolated EPROTO). Test pravděpodobně selhal.

                  
suppressedRučně skryté manuálně přes API (pro odfiltrování známých falešných alarmů).

                
              

            


            
            

              
6. Časté scénáře — co znamenají v praxi


              

                
Vidím jen command_chunk a žádný command

                
Pravděpodobně se posunula tabulka mimo začátek. Hledej parent command s ⛓ markerem výš (má stejné cmd_name). Continuation chunky jsou italic / dimmed.

              


              

                
Po command následuje incomplete_segment a žádný response_ok

                
Tester poslal příkaz, ale než dorazila odpověď, zaslal další příkaz nebo nastal reconnect. Předchozí příkaz nemá známou odpověď — buď ho zařízení nezpracovalo, nebo odpověď spadla. Podívej se kauzální kontext (panel vpravo) — typicky to bývá v důsledku timeout nebo reconnect těsně před tím.

              


              

                
Vidím timeout, pak response_error u dalšího příkazu

                
Klasický „domino efekt": zařízení se zaseklo na předchozím příkazu, časem se vzpamatovalo, ale interní stav je rozbitý → další příkaz selhává. Detail vpravo by měl ukázat kauzální hint typu „Timeout na X před chybou mohl rozbít stav zařízení".

              


              

                
Spousta crc_probe a ERROR invalid-crc

                
To je záměrné. Tester při startu zkouší jestli je CRC mód aktivní. crc-enable bez CRC suffixu → zařízení vrátí ERROR invalid-crc = ano, CRC běží. Severity je info, ne critical. Pokud chceš schovat, zaškrtni hide probes.

              


              

                
Mám 2 device sessions ve Sessions tabu

                
Capture obsahuje výměnu zařízení nebo reassignment USB adresy (typicky po reconnect). Každá session má vlastní serial v OK XXXXXXXX odpovědích. device_change meta event ve Flow označuje hranici.

              


              

                
Latence příkazu skočila z 10 ms na 300 ms

                
Detektor to označí jako timing_critical. Možné příčiny: zařízení provádí dlouhou operaci (kryptografii, flash write), nebo kontentaci na USB sběrnici. Zkontroluj response_progress řádky — pokud přichází # hlášení, zařízení něco dělá; pokud je ticho, je to anomálie.

              


              

                
Příkaz má vedle sebe ikonu ⛓

                
Chunkovaný příkaz — text byl tak dlouhý, že se nevešel do jednoho USB paketu a USB ho rozdělil. Parent řádek má kompletní obsah po složení všech chunků (zobrazený s prefixem [chunked: N pkts, M B]); pod ním následují command_chunk řádky (italic / dimmed) — pokračovací pakety. Není to chyba.

              


              

                
Hledám konkrétní DUT (kus HW)

                
Otevři záložku Sessions, klikni Načíst sessions a runy, pak do pole Hledat DUT SN piš část seriálu. Filtr je case-sensitive (rozlišuje velká/malá písmena), podporuje /regex/. Klik na řádek tě skočne na první událost tohoto DUTu ve Flow.

              


              

                
„Server ztratil aktuální capture — nahrajte znovu"

                
Po restartu serveru se ztratila mapa capture-id → soubor. Aplikace automaticky resetuje stav a zobrazí upload zónu. Nahraj PCAP znovu. Server se snaží mapu perzistovat na disk (USB_ANALYSIS_STATE_DIR), ale dočasné PCAP soubory v /tmp systém čas od času promazává — pak je třeba je nahrát znovu.

              


              

                
Ve Flow analyzeru nevidím normální command/response řádky

                
Severity filter je výchozím nastavený na warning+, který schovává běžné ok události (úspěšné commandy + OK odpovědi). Přepni na ALL v Severity selectoru pro kompletní výpis.

              


              

                
Skok na konkrétní řádek

                
Stiskni G (jako „goto"), zadej seq číslo, Enter. Funguje napříč filtry. Funguje i s plně načteným streamem (žádné lazy fetching) — skok je instantní.

8. Limity uploadu a konfigurace

              
Aktuální hodnoty server-side limitů uvidíš v About modálu (klik na USB Analysis v levém horním rohu hlavičky → sekce „Limity uploadu"). Výchozí hodnoty:

              

                
LimitVýchozíEnv proměnná

                
                  
Velikost jednoho souboru2 GBUSB_ANALYSIS_MAX_UPLOAD_BYTES

                  
Počet souborů v jednom uploadu64USB_ANALYSIS_MAX_UPLOAD_FILES

                  
Souhrnný limit multi-uploadu16 GBUSB_ANALYSIS_MAX_UPLOAD_TOTAL_BYTES

                  
Flow LRU cache (zanalyzované captures)16 záznamůUSB_ANALYSIS_FLOW_CACHE_MAX

                  
Perzistence capture-id mapy$TMPDIR/usb-analysisUSB_ANALYSIS_STATE_DIR

                
              

              

                Pozor na reverse proxy. Aplikace běžící za nginx / Cloudflare / Traefik nasazením může narazit na body-size limit proxy (často 1 MB - 100 MB). Pak se velký upload uřízne dřív, než dojde do FastAPI. Konfigurace závisí na deployi — viz README sekce „Reverse proxy".
              

              
Po restartu serveru se mapa capture-id obnoví z disku (captures.json v STATE_DIR), pokud dočasné PCAP soubory v /tmp stále existují.

            


            


            


            
            

              
1. Quick start

              
    
                
  1. Upload a PCAP. Use Upload in the left sidebar or drop files onto the upload area. The app accepts .pcap, .pcap00–.pcapNN in LINKTYPE_USB_LINUX_MMAPPED format (Linux usbmon capture).
    2. 
                
  2. After upload a green dot lights up in the header and the Overview tab shows the summary (packet count, duration, sessions, runs, critical events).
    3. 
                
  3. For a detailed view switch to the Flow analyzer tab — the main view with chronological commands and responses.
    4. 
                
  4. If something failed, open the Errors tab to see all detected problems. Clicking a row jumps to its location in Flow.
    5. 
                
  5. For CI integration grab a JUnit XML from the Export tab.
    6. 
                
  6. Version + limits: click USB Analysis in the top-left corner — a modal shows current version, upload limits and server runtime.
    7. 
              

            


            
            

              
2. What each tab does

              

                
TabPurpose

                
                  
OverviewCapture summary: packet count, duration, runs, sessions, critical/warning counts. First stop after uploading.

                  
PacketsRaw USB packet table (technical view). Each row = one URB Submit/Complete with metadata (bus, device, endpoint, payload hex).

                  
StreamWireshark-style "Follow Stream" — ASCII translation of bulk communication. Shows what the tester sent and what the device replied.

                  
Flow analyzerMain view: chronological command ↔ response timeline with timeout / CRC / reconnect detection and causal hints. Click a row to view its JSON on the right.

                  
ErrorsFiltered list of problems only (timeouts, CRC mismatch, EPROTO …). Click = jump to Flow.

                  
SessionsPer-device (bus/dev/serial) and per-run statistics. Useful when a capture spans device swaps or multiple test runs.

                  
Deep analysisMachine learning: comparison against a learned baseline, rule mining across files. Long operation — must be triggered manually with the button.

                  
ExportDownload the flow analysis as JSON / CSV / HTML / JUnit XML.

                  
HelpThis window — manual, event-class glossary and keyboard shortcuts. Available without an uploaded capture.

                
              

            


            
            

              
3. Reading the Flow analyzer

              
Each row is one event. Columns from left to right:

              

                
ColumnMeaning

                
                  
seqEvent ordinal within the capture. Key for cross-references and pairing.

                  
delta_msTime since the previous event in milliseconds. A large jump = potential stall.

                  
directionHOST→DEV tester sent data; DEV→HOST device responded; INTERNAL meta event (timeout, reconnect…).

4. Event-class glossary

              
Every row in Flow carries a tag like [command], [timeout], [incomplete_segment]. Here's what they mean:


              
Tester SN vs. DUT SN — who's who

              

                
TermMeaning

                
                  
Tester SNProgramming-station (tester) serial. Comes from OK XXXXXXXX responses (8 hex chars). Constant across all runs in one capture — one tester programs many DUTs.

                  
DUT SNSerial of the specific hardware piece (Device-Under-Test). Comes from the checked-otp-device-sn-write <SN> command — the first argument after the command name. Changes per run — each run programs a new DUT.

                  
device_serial in event JSONCurrent identity: DUT SN takes priority (once written in a run), otherwise tester SN. This is "who is talking right now" from the user's viewpoint.

                
              


              
Communication events

              

                
ClassMeaning

                
                  
commandTester sent a command to the device (e.g. ping, checked-optiga-init). Row content is the full sent text including the CRC suffix.

                  
command_chunkContinuation packet of a long command (e.g. checked-secrets-certdev-write with a large certificate doesn't fit in a single USB packet and is sent in pieces). Not an error — purely a visual reflection of the physical split.

                  
crc_probeTester probes whether CRC mode is active. Deliberately sends a command without CRC. If the device replies ERROR invalid-crc, it means CRC is active (= probe succeeded).

                  
response_okDevice replied OK or OK NO. OK NO = operation completed but the result is negative (e.g. lock-check says "unlocked").

                  
response_errorDevice replied ERROR. Always critical (except for crc_probe replies, which are expected).

                  
response_progressProgress status line (starting with #) — informational, not the operation outcome.

                  
response_dataOther device data not starting with OK, ERROR or #.

                
              


              
Detected problems

              

                
ClassMeaning

                
                  
timeoutTester sent a command and got no reply within 30 s. Either the device hung or the response didn't reach the host. Content shows actual latency (e.g. Timeout 35000.0ms on checked-optiga-init).

                  
incomplete_segmentPrevious command wasn't closed when a new one arrived. Example: tester sent command A, then B before A's reply, or a reconnect / device_change happened in between. The communication state is inconsistent — the device may or may not have processed A. Note: a continuation packet of a chunked command (parent without CR/LF) is detected as command_chunk, not incomplete_segment — chunking is not an error. But if a chunked command is interrupted by device_change or timeout, incomplete_segment IS emitted because the reply will never arrive.

                  
reconnectThe gap between two bulk transfers exceeded 5 s — the device likely re-enumerated or restarted. Usually followed by a sequence of control transfers (re-enumeration).

                  
usb_errorUSB physical-layer error (e.g. status -71 = EPROTO = protocol error, signal integrity, DN/DP). When followed by reconnect, typically a device reset (info severity); isolated EPROTO without reconnect = critical.

                  
lost_urbA USB Request Block was Submit-ted but no Complete arrived within 60 s. The message was "lost" in the driver or device.

                  
device_changeA packet appeared with a different (bus, device_address) tuple — typically a device swap or address reassignment after reconnect. One device session ends and another begins here.

                
              


              
Tags in the Errors table (after detectors)

              

                
event_typeMeaning

                
                  
missing_crcA checked-* command was sent without a CRC suffix (8 hex chars at the end). In CRC mode the device rejects it.

                  
crc_mismatchThe CRC suffix doesn't match the computed CRC32 of the command body. Either tester bug or bus corruption.

                  
app_errorApplication layer: device returned ERROR ….

                  
lock_regressionAfter a successful checked-*-lock, the following lock-check returned OK NO (= "unlocked"). Something unlocked itself when it shouldn't.

                  
timing_high / timing_criticalCommand latency >5× / >15× higher than the median of its other invocations. The device is slowing something down.

                  
timing_suspiciously_lowLatency <50% of the lowest observed minimum. Possibly a short-circuit in logic or a cached response.

                  
usb_eproto_isolatedEPROTO error not followed by reconnect — likely a permanent hardware issue.

                  
usb_eproto_reconnectEPROTO error followed by reconnect — typically a device reset after watchdog timeout.

                  
usb_lost_urbSee above — Submit without Complete within the 60 s window.

                  
device_reconnectDevice reconnect (may be expected or unexpected depending on test context).

                
              

            


            
            

              
5. Severity levels

              

                
LevelMeaning

                
                  
okNormal state (successful response, valid command). Hidden under the default warning+ filter.

                  
infoInformative (probe, progress, expected reconnect). Not a problem but worth noting.

                  
warningSomething's not ideal (OK NO, higher latency, lost URB). Communication works but you should investigate.

                  
criticalDefinite problem (timeout, ERROR, missing CRC, isolated EPROTO). Test most likely failed.

                  
suppressedManually hidden via API (for filtering out known false alarms).

                
              

            


            
            

              
6. Common scenarios — what they mean in practice


              

                
I only see command_chunk and no command

                
You're likely scrolled past the start. Look above for the parent command with the ⛓ marker (same cmd_name). Continuation chunks are italic / dimmed.

              


              

                
After a command comes incomplete_segment and no response_ok

                
The tester sent a command but before its reply arrived, another command was sent or a reconnect happened. The previous command has no known reply — either the device didn't process it or the reply dropped. Check the causal context (right panel) — usually you'll find a timeout or reconnect just before.

              


              

                
I see timeout then response_error for the next command

                
Classic "domino effect": the device hung on the previous command, eventually recovered, but its internal state is corrupted → the next command fails. The right detail panel should show a causal hint like "Timeout on X before the error may have corrupted device state".

              


              

                
Lots of crc_probe and ERROR invalid-crc

                
This is intentional. At startup the tester probes whether CRC mode is active. crc-enable without CRC → device returns ERROR invalid-crc = yes, CRC is on. Severity is info, not critical. To hide them, tick hide probes.

              


              

                
I have 2 device sessions in the Sessions tab

                
The capture contains a device swap or USB-address reassignment (typically after reconnect). Each session has its own serial in OK XXXXXXXX responses. The device_change meta event in Flow marks the boundary.

              


              

                
Command latency jumped from 10 ms to 300 ms

                
The detector tags this as timing_critical. Possible causes: device doing a long operation (crypto, flash write), or USB bus contention. Check response_progress rows — if # messages arrive, the device is working; if it's silent, that's an anomaly.

              


              

                
A command has a ⛓ icon next to it

                
Chunked command — the text was too long to fit in a single USB packet and got split by USB. The parent row contains the fully reassembled content (with a [chunked: N pkts, M B] suffix); below it follow command_chunk rows (italic / dimmed) — the continuation packets. Not an error.

              


              

                
I'm searching for a specific DUT (HW piece)

                
Open the Sessions tab, click Load sessions and runs, then type part of the serial into Search DUT SN. The filter is case-sensitive, supports /regex/. Click a row to jump to the DUT's first event in Flow.

              


              

                
"Server lost the current capture — please re-upload"

                
After a server restart the capture-id → file map is gone. The app automatically resets state and shows the upload zone. Upload the PCAP again. The server attempts to persist the map to disk (USB_ANALYSIS_STATE_DIR) but the temporary PCAP files under /tmp get cleared periodically by the OS — then they must be re-uploaded.

              


              

                
I can't see normal command/response rows in Flow analyzer

                
The severity filter defaults to warning+, which hides everyday ok events (successful commands + OK responses). Switch to ALL in the Severity selector to see the full stream.

              


              

                
Jump to a specific row

                
Press G (as in "goto"), type the seq number, Enter. Works across filters. Works against a fully-loaded stream (no lazy fetching) — the jump is instant.
